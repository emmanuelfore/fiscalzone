# ZIMRA-Compliant Invoicing SAAS Application - Complete Development Prompt

## Project Overview
Create a complete multi-tenant SAAS invoicing application for Zimbabwe businesses that is fully compliant with ZIMRA (Zimbabwe Revenue Authority) regulations. The application must integrate with the FDMS (Fiscal Data Management System) API and generate QR code-enabled fiscal invoices and quotations.

## IMPORTANT: Use Official ZIMRA FDMS API
**API Documentation:** https://fdmsapitest.zimra.co.zw/swagger/index.html
- All FDMS integrations MUST follow the exact specifications from this official Swagger documentation
- Review all endpoints, request/response schemas, and authentication methods from the API docs
- Implement error handling based on the official error codes and messages

## Tech Stack Requirements
- **Frontend**: Next.js 14+ (App Router), React 18+, TypeScript
- **Styling**: Tailwind CSS with shadcn/ui components
- **Backend**: Next.js API Routes (serverless functions)
- **Database**: Supabase (PostgreSQL) with Prisma ORM
- **Authentication**: Supabase Auth with email/password and social login (Google)
- **File Storage**: Supabase Storage for logos and documents
- **QR Code Generation**: qrcode.react library
- **PDF Generation**: @react-pdf/renderer or jsPDF
- **Email Service**: Resend or SendGrid for transactional emails
- **Payment Processing**: Stripe for subscription management
- **State Management**: React Context API and React Query (TanStack Query)
- **Form Handling**: React Hook Form with Zod validation
- **Date Handling**: date-fns
- **Animations**: Framer Motion for landing page
- **Icons**: Lucide React
- **Deployment**: Optimized for Replit deployment

**Critical FDMS Functions:**

```typescript
// /lib/fdms.ts

interface FDMSConfig {
  deviceId: string;
  apiKey: string;
  certificate?: string;
  baseUrl: string; // ZIMRA FDMS API endpoint
}

interface FDMSInvoiceData {
  invoiceNumber: string;
  issueDate: string;
  customerTIN?: string;
  customerName: string;
  items: Array<{
    description: string;
    quantity: number;
    unitPrice: number;
    taxRate: number;
    taxAmount: number;
    total: number;
  }>;
  subtotal: number;
  totalTax: number;
  total: number;
  currency: string;
}

interface FDMSResponse {
  success: boolean;
  fiscalCode?: string;
  fiscalDayNumber?: number;
  fiscalReceiptNumber?: string;
  qrCodeData?: string;
  timestamp?: string;
  errorCode?: string;
  errorMessage?: string;
}

class FDMSService {
  // Open fiscal day (required before issuing invoices)
  async openFiscalDay(config: FDMSConfig): Promise<FDMSResponse>
  
  // Close fiscal day (end of business day)
  async closeFiscalDay(config: FDMSConfig): Promise<FDMSResponse>
  
  // Submit invoice to FDMS
  async submitInvoice(
    config: FDMSConfig,
    invoiceData: FDMSInvoiceData
  ): Promise<FDMSResponse>
  
  // Verify fiscal code
  async verifyFiscalCode(
    config: FDMSConfig,
    fiscalCode: string
  ): Promise<FDMSResponse>
  
  // Get device status
  async getDeviceStatus(config: FDMSConfig): Promise<FDMSResponse>
  
  // Cancel/void invoice in FDMS
  async voidInvoice(
    config: FDMSConfig,
    fiscalCode: string,
    reason: string
  ): Promise<FDMSResponse>
}
```

**FDMS Integration Requirements:**
- Automatic submission to FDMS when invoice is issued
- Retry logic for failed submissions
- Queue system for offline invoice creation (sync when online)
- FDMS day management (open/close)
- Error handling with user-friendly messages
- Fiscal code storage and verification
- Compliance validation before submission
- Audit trail of all FDMS communications
